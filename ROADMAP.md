# Roadmap

Axon's development has proceeded in informal, numbered phases (see git history and inline comments for Phases 1–7: CS API core, federation foundation + state-res v2, E2EE sync, media/push/app-services, spaces/relations/room upgrades, delegated OIDC login, MSC3814 dehydrated devices). This document tracks what's planned from Phase 8 onward.

## Phase 8 — E2EE reliability & cross-server delivery (done)

Despite a substantial existing E2EE surface (key upload/query/claim, cross-signing, key backup, dehydrated devices), three relay-layer bugs made device verification and message delivery unreliable across clients and homeservers:

- `sendToDevice` never woke a long-polling `/sync` — a queued Megolm room-key share or SAS verification message sat undelivered until the recipient's client-chosen timeout happened to elapse on its own, not until the sender actually sent it. Every existing sync test used `timeout=0`, so this was invisible to the suite.
- `device_lists.changed` only fired on key upload/cross-signing, never on newly sharing a room — a client trusting that signal (rather than deriving key-query needs from room membership itself) would never learn to query/verify a fresh room-mate's devices. `device_lists.left` was hardcoded to `[]`.
- `sendToDevice` had no local/remote split, unlike `/keys/query` and `/keys/claim` — any to-device message aimed at a user on another homeserver was silently dropped forever, since there was no federation EDU handling at all (inbound or outbound).

Fixed by: broadcasting a PubSub wake on to-device delivery (mirroring the existing `device_list_updates` pattern); re-touching `device_list_updates` for both parties on room join and a new `device_list_partings` table + cursor for room leave; and adding `m.direct_to_device` EDU support (outbound via `AxonWeb.FederationFanout`, inbound via `FederationController.send_transaction/2`), plus wildcard (`"*"`) device-id expansion. Regression coverage in `apps/axon_web/test/e2ee_delivery_test.exs` exercises the real long-poll path with a nonzero timeout (closing the test-coverage gap that hid bug 1) and the cross-server EDU path via `AxonFederation.FakeRemoteMatrixServer`.

## Phase 9 — Federation EDU & real-time parity (done)

Full bidirectional EDU support (`m.typing`, `m.receipt`, `m.presence`), building on the `m.direct_to_device` scaffolding from Phase 8, plus durable outbound federation delivery.

- **Typing indicators** (`PUT /rooms/:room_id/typing/:user_id`) were a complete no-op stub — acknowledged and discarded. Now backed by `AxonSync.Typing` (in-memory, auto-expiring, mirroring `AxonSync.Presence`'s design), surfaced in `/sync`'s per-room `ephemeral` section, and federated as `m.typing` EDUs both ways.
- **Read receipts** had no PubSub wake at all (unlike to-device/device-lists/account-data, fixed in Phase 8) — a receipt posted to an otherwise-quiet room sat unseen until something else happened in that room, since incremental sync only included a room at all when it had a *new timeline event*. Fixed with a new `ephemeral_updates` log (mirroring `device_list_updates`) that both wakes long-pollers and makes a room eligible for inclusion on ephemeral-only changes; `m.read` receipts (never `m.read.private`) now federate as `m.receipt` EDUs.
- **Presence** now federates as `m.presence` EDUs to every server sharing a room with the user, but only on an actual state transition (online/unavailable/offline or status_msg change) — not on every `bump_activity` touch, which happens on every authenticated request and would otherwise flood federation peers.
- **Durable outbound delivery**: `AxonFederation.OutboundQueue` persists every outbound PDU/EDU transaction before the first attempt and retries with exponential backoff (30s → 1hr cap, giving up after 7 days) on failure, reusing the same row id as the txn_id on every retry so a remote server that already processed an earlier attempt responds idempotently. Replaces the previous fire-and-forget behavior in `AxonWeb.FederationFanout`, where a failed delivery just logged a warning and was dropped — meaning a remote server being briefly unreachable silently lost whatever was sent to it during that window. Matters here specifically because this deployment federates with other servers in production use, not just same-server accounts.

Regression coverage: `apps/axon_federation/test/outbound_queue_test.exs` (failure → persisted → retry → success), `apps/axon_web/test/phase9_ephemeral_test.exs` (typing/receipt local wake-up and room-inclusion, both EDU directions for typing/receipts/presence, permission checks, expiry).

## Phase 10 — Sliding Sync (MSC4186) (done)

`POST /_matrix/client/unstable/org.matrix.msc4186/sync` alongside classic `/sync` (kept for legacy clients). A pragmatic subset of the MSC rather than full conformance:

- **Lists**: `ranges`-paginated, recency-sorted (`sort` is always treated as `by_recency`) room lists over the user's *joined* rooms, with `is_dm`/`is_encrypted` filters. Every response re-sends a full `SYNC` op per range rather than diffing against a remembered window — no `conn_id` session state is kept, so this is spec-valid but less bandwidth-efficient than real add/remove diffing.
- **`room_subscriptions`**: explicit per-room overrides, included regardless of list ranges; merged with any list config referencing the same room (union of `required_state`, max `timeline_limit`).
- **`required_state` resolution**: concrete `[type, state_key]` pairs, `*` wildcards, `$ME` (substitutes the requesting user), and `$LAZY` (member events lazy-loaded to just the current timeline's senders, plus self) — reusing `EventStore.get_current_state/1`.
- **Extensions**: `to_device`, `e2ee` (OTK/fallback-key counts, `device_lists`), `account_data` (global + per visible room), `receipts` (`m.read` only), `typing` — all built from the same `AxonWeb.SyncHelpers` module classic `/sync` now delegates to as well, so the two endpoints can't drift apart on E2EE/device-list/ephemeral semantics (the extraction this required also served as a small refactor: those cursor/extension helpers used to be private duplicates baked into `SyncController`). Extension params aren't sticky across requests — a client must resend `"enabled": true` every time, not just once.
- **Known gaps, deliberately deferred**: invited/knocked/left rooms don't appear in `lists`/`room_subscriptions` yet (exposing them safely means re-deriving the same invite-state-only visibility rules classic sync's `build_invite_state/2` already enforces — deferred rather than done half-safely); `notification_count`/`highlight_count` are always 0 (this codebase doesn't compute per-room unread/highlight counts anywhere yet, including in classic `/sync` or push dispatch, so this isn't a regression).

Long-poll wake-up reuses `AxonSync.Manager.wait_for_events/3` unchanged, so the Phase 8 wake-up fixes (to-device, device-list, ephemeral) apply here too. Regression coverage: `apps/axon_web/test/sliding_sync_test.exs` (13 tests: list sort/ranges/filters, `room_subscriptions`, `required_state` incl. `$LAZY`/`$ME`, the long-poll wake path with a real nonzero timeout, and every extension).

## Phase 11 — E2EE/verification test hardening (done)

Two real bugs found and fixed while expanding coverage of the surface that's seen the most iterative bugfixing:

- **Stale test, not a stale bug**: `e2ee_multi_device_test.exs` asserted a first-time cross-signing `master_key` upload returns 401 requiring a UIA retry. MSC3967's `uia_exempt?/4` (`key_controller.ex`) correctly exempts first-time setup (nothing on file yet to protect) and has for a while — the test just never caught up. Fixed the assertion and added a real UIA-required case alongside it (rotating to an unrelated master key that isn't signed by the one on file), so the branch that *does* still require UIA has coverage too.
- **Key backup consistency**: `PUT /room_keys/keys` never validated its `version` query param against the user's actual current backup version — a client retrying against a stale, deleted, or superseded version would silently insert rows with no error, and `count`/`etag` on `GET /room_keys/version` would never reflect the write. Now returns `403 M_WRONG_ROOM_KEYS_VERSION` (with `current_version` in the body) per spec, covered by `apps/axon_web/test/key_backup_consistency_test.exs`.

New Complement-style scenario coverage in `apps/axon_web/test/e2e/verification_flow_test.exs`:

- A full `m.key.verification.*` SAS exchange (request → ready → start → key ×2 → mac ×2 → done) between two users' devices, with every hop driven through a real nonzero-timeout long-poll on both sides — not a single event checked with a short poll, which is all any prior test did.
- The same shape across two homeservers via `AxonFederation.FakeRemoteMatrixServer`, with a realistic nonzero timeout on the inbound EDU recipient's long-poll — the existing federation EDU test only short-polled after the fact, so it couldn't catch a wake-up regression on the federated path the way it already could locally.
- The wildcard `"*"` to-device target (`KeyStore.expand_wildcard_device/2`, added in Phase 8) had zero test coverage until now — verified against a 3-device account, confirming delivery to all three and non-delivery to an unrelated user.
- A cross-signing trust-chain scenario: bob signs alice's master key after "verifying" her; `/keys/query` shows bob his own signature but not an uninvolved third party, exercising the visibility rule end-to-end over HTTP rather than just at the `KeyStore` unit level.

## Phase 12 — Room version 12, third-party invites, guest access (done)

Turned out to be a much bigger lift than the original one-line scope suggested: room v12 (MSC4297 + MSC4289) isn't just a state-res tweak, it also changes the room ID format and the power-level model. Implemented for real rather than descoped, since a partial version-12 room would be actively misleading (claiming compliance while silently breaking on the very things that changed).

**Room version 12** (`AxonRoom.CreateRoom`, `AuthRules`, `EventBuilder`, `StateResV2`):
- **Domainless, hash-derived room IDs**: a v12 room's ID is its own `m.room.create` event's ID (reference hash) with `!` instead of `$` — no `:server_name` suffix at all. This is a real chicken-and-egg problem (the room "exists" — DB row, `RoomProcess` — only after the create event is built and hashed, but every other creation step needs the room_id first): `CreateRoom.execute/2` now builds+hashes the create event standalone (`EventBuilder.build/4` with `room_ctx.room_id: nil` omits the `room_id` field entirely, matching the wire requirement that the create event MUST NOT carry one), derives `room_id` from its hash, *then* creates the room row and injects the pre-built event via `RoomProcess.apply_remote_event/2`. `EventStore.event_to_map/1` omits `room_id` specifically for a v12 create event on every outbound path (sync, federation `/state`, `/send_join`, transactions) since it's the single canonical serialization function. Room upgrades to v12 reverse their usual order (new room created first, *then* the old room is tombstoned with the now-known new room_id) since the new ID can't be pre-generated the way every earlier version's `!random:server` id could.
- **Creator privilege (MSC4289)**: a v12 room's "creators" (the create event's `sender` plus any `additional_creators` from its content, passed like any other `creation_content` field) get implicit, infinite power — never listed in `power_levels.users` (a `power_levels` event that lists one is rejected, rule 10.4), can't be outranked or demoted. `additional_creators` is validated as an array of real user IDs at both room-creation time and on any inbound create event.
- **Auth-events selection**: `m.room.create` is excluded from every other event's `auth_events` in v12 (rule 3.2) — fixed in both the local build path (`EventBuilder`) and the federation join/knock/leave template path (`FederationController`), which duplicates that selection logic since remote joins don't go through the normal local build.
- **State-res v2.1** (`AxonRoom.StateResV2`): the one piece that actually matches the original one-line scope — iterative auth checks now start from an empty state map for v12 instead of the unconflicted state map, per MSC4297. The deeper "conflicted state subgraph"/"auth difference over the full conflicted set" refinement isn't implemented; `AxonRoom.StateResolver` (this module's only caller) already deliberately resolves just enough state for one incoming PDU's auth check rather than replaying arbitrary historical DAG forks, so that refinement isn't reachable through the current call path regardless.
- **Defensive fix**: joining/knocking a domainless room_id with no `via`/`server_name` hint used to silently fall back to treating the whole room_id string as a server name (garbage). Now returns a clear `M_MISSING_PARAM` instead.

Regression coverage: `apps/axon_web/test/room_v12_test.exs` (creation, creator/additional_creators power, rule 10.4 rejection, non-creator power gating still works, federation join of a v12 room via `FakeRemoteMatrixServer`, the via-hint error).

**Third-party (3pid) invites** (`m.room.third_party_invite`, `AuthRules.valid_third_party_invite?/3`): `POST /rooms/:roomId/invite` now has a real `medium`+`address` variant that creates an `m.room.third_party_invite` state event (rule 7: gated by invite power) with a token and this server's own signing key, rather than a direct `m.room.member` invite (impossible — no mxid is known yet). `POST /join/:roomId` accepts an optional `third_party_signed` proof, which — if its `signed.token` matches a live invite event in the room and its signature verifies against that event's `public_key`(s) — authorizes the join regardless of the room's normal `join_rule`, exactly like a direct invite would.

Known, deliberate gap: axon has no identity-server integration, so nothing actually delivers a 3pid invite to the invitee (no email/SMS sent) — the room-state mechanics and cryptographic proof are fully real and spec-shaped (self-signed with this server's own key, since there's no external identity server to delegate to), but getting the token to the invitee is left to an out-of-band channel until real mail/identity integration exists. `PUT federation/v1/exchange_third_party_invite` (for a remote server to ask a room-resident server to mint the invite event on its behalf) is not implemented — it would only be reachable once an external identity server both sides trust exists, which axon doesn't provide, so it'd be dead code today.

Regression coverage: `apps/axon_web/test/third_party_invite_test.exs` (invite creation, power gating, valid-proof join, mxid-mismatch rejection, bad-signature rejection).

**Guest access**: audited against spec. The one protocol-mandated, testable guest-specific behavior — `m.room.guest_access` gating on join — was already implemented and tested (`apps/axon_web/test/e2e/guest_access_flow_test.exs`, pre-existing). Didn't find another well-established *protocol* requirement worth adding; further restriction beyond that is a server-policy choice, not a spec requirement, and guessing at additional restrictions risked incorrectly blocking guests from things they're actually allowed to do. No changes made.

## Phase 13 — Operability: admin API & rate limiting (done)

A real admin API under `/_synapse/admin/v1` (mirroring the naming convention the existing shared-secret registration bootstrap already used), gated by a new `AxonWeb.Plug.RequireAdmin` checking `users.admin` — which turned out to be a pre-existing, silently-toothless field: the `admin` flag accepted by the Synapse-compatible bootstrap registration endpoint was parsed and cryptographically bound into its MAC check, but never actually passed through to `UserStore.register/3`, so no account could ever really become an admin before now. Fixed as part of wiring the gate up (the alternative — building an admin API nothing could ever reach — wouldn't have been shippable).

- **User management**: `GET /users` (paginated, filterable by guest/deactivated status), `GET /users/:id`, `POST /deactivate/:id` (admin-initiated; refactored the existing self-service `/account/deactivate` to share the same `UserStore.deactivate/1`), `POST`/`DELETE /users/:id/shadow_ban`.
- **Shadow-ban enforcement** (not just a flag): a shadow-banned user's own non-state (message-like) events are invisible to every *other* local viewer's `/sync` — both classic and sliding — but always visible to themselves (reading your own writes must still work), and never federated out. State events (joins, etc.) stay visible everywhere regardless; hiding those would corrupt other viewers'/servers' picture of room membership, a bigger inconsistency than muting a spammer's messages. Enforced in `EventStore.get_user_events_since/2` and `get_recent_room_events/3` (local sync fan-out) and `RoomProcess`'s federation broadcast (outbound).
- **Room management**: `GET /rooms`, `GET /rooms/:id`, `DELETE /rooms/:id` — a real purge (deletes the room's events/state/memberships/account-data/receipts from local storage) that also marks the room `blocked` so a future join attempt gets a clear, specific rejection instead of silently doing nothing; also terminates the room's live process if one is resident in memory, so it can't keep serving stale state after the rows it was built from are gone.
- **Media quarantine**: `POST`/`DELETE /media/quarantine/:server/:id` — quarantined media is served as a plain 404 (matches Synapse), not a distinct error, so a client/scraper can't use the response shape to tell "never existed" apart from "admin pulled it".
- **Report-review queue**: `GET /event_reports`, paginated — reports were already being collected via the client `POST /rooms/:roomId/report[/:eventId]` endpoints but nothing had ever surfaced them until now.
- **Rate limiting**: a small self-contained ETS-backed sliding-window limiter (`AxonWeb.RateLimiter`, mirroring `AxonSync.Typing`'s GenServer+ETS pattern rather than adding a new dependency for what's a contained need), applied to login (per-IP), registration (per-IP), and message send (per-user) via a controller-level plug. Limits are runtime-configurable (`config :axon_web, :rate_limits`), which is what let `config/test.exs` set them high enough that the test suite's own traffic (many registrations/logins/sends from the same loopback IP across many files) never trips them.

Regression coverage: `apps/axon_web/test/admin_controller_test.exs` (the `RequireAdmin` gate, every admin endpoint), `apps/axon_web/test/shadow_ban_test.exs` (local sync filtering both directions, state-event visibility, federation skip via `FakeRemoteMatrixServer`), `apps/axon_web/test/rate_limit_test.exs` (the limiter directly, and the login plug's 429 shape end-to-end).

## Phase 14 — Feature completion pass (done)

The roadmap's last phase — Phase 8 through here is now the complete arc originally planned.

- **Push rules persistence**: `PUT`/`DELETE`/`.../enabled`/`.../actions` on `/pushrules/...` used to silently discard — only the hardcoded server default rule set was ever actually served or evaluated, regardless of what a client tried to customize. Now backed by `AxonPush.UserRules`: a server-default rule (`.m.rule.*`) can have its `enabled`/`actions` overridden but not its `conditions`/`pattern` (fixed, per spec); any other rule id is a genuine custom rule, evaluated before the (possibly-overridden) defaults for its kind. `AxonPush.Dispatcher` now evaluates each recipient's actual merged ruleset instead of always the defaults. Fixed a real bug found in the process: `room`/`sender` kind rules were falling through to generic conditions-based matching, which vacuously matched *every* such rule against *every* event (an empty `conditions` array trivially satisfies `Enum.all?`) — they were simply unreachable before now since no default `room`/`sender` rules exist, but would have misfired the instant a user's custom one became reachable. Not implemented: the `before`/`after` positioning query params — custom rules are appended in creation order within their kind, a contained gap (affects only tie-break ordering among a user's own custom rules, not whether notifications fire).
- **Server notices**: a lazily-provisioned system account (`@server-notices:<server_name>`) an admin can use (`POST /_synapse/admin/v1/send_server_notice`) to push a message into a dedicated room with any local user, auto-created and auto-joined (not just invited — an invite-only membership can't see timeline content, which would defeat the purpose of a notice) on first use and reused for every subsequent notice to that user, tagged `m.server_notice` in their room account data so a real client can render it distinctly from an ordinary DM.
- **SSRF-safe URL previews**: `GET /_matrix/client/v1/media/preview_url` (previously a deliberate 404) now really fetches and parses OpenGraph metadata, with layered SSRF defense — scheme allowlist, literal-IP and resolved-DNS rejection of private/loopback/link-local/multicast/reserved ranges (IPv4 and IPv6, including IPv4-mapped IPv6), manually-followed and re-validated redirects (capped hops), and response size/time limits. An `og:image`, if present, is fetched through the same hardened path and rehosted as local media so the response returns an `mxc://` URI rather than leaking the original image URL. Results are cached (1hr TTL). Known, documented limitation: not immune to DNS rebinding (the resolve-and-validate check and the HTTP client's actual connection are two separate steps) — closing that fully needs pinning the connection to the specific validated address, not implemented here; this still blocks the overwhelming majority of real-world SSRF payloads.

Regression coverage: `apps/axon_web/test/push_rules_controller_test.exs` + `apps/axon_web/test/e2e/push_pipeline_flow_test.exs` (a muted room's push rule actually suppresses delivery end to end), `apps/axon_web/test/server_notices_test.exs`, `apps/axon_media/test/url_preview_test.exs` (SSRF blocking and OpenGraph parsing — tested separately by design: real network calls are asserted *rejected* rather than faked, and parsing is exercised directly against canned HTML) + `apps/axon_web/test/media_controller_test.exs`.
